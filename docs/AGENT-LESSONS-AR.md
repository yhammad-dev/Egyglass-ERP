# دروس التعامل مع وكلاء البناء (OpenCode)
# Agent Interaction Lessons — EgyGlass ERP

> هذا الملف يوثّق ما تعلّمناه من تشغيل أول وكيل بناء (Stream A).
> الهدف: تفادي الأخطاء المتكررة، وجعل الوكلاء أوضح في تحليل الجذر.
> المبدأ الحاكم: **بالدليل لا بالحدس** — ادعاء الوكيل ليس دليلًا؛ الدليل ما نراه بأنفسنا.

---

## القسم 1 — أخطاء الوكيل التي اكتشفناها وكيف نتفاداها

### خطأ 1: الوكيل يدّعي "BUILD GREEN" والبناء أحمر
**ما حدث:** الوكيل قال "build passes cleanly" مرتين، والخطأ (سطر 611) لا يزال موجودًا.
**السبب:** الوكيل يفترض النجاح دون تشغيل build حقيقي، أو يعتمد على نتيجة قديمة.
**القاعدة:**
- لا تثق بكلمة "GREEN". اطلب **ناتج الـ build الكامل الخام** (raw, unedited).
- الأفضل: **أنتَ** تشغّل الـ build بنفسك — لا وسيط يدّعي:
  `docker compose exec -e NODE_ENV=production app npm run build`
- لا تعتمد قط checkpoint بناءً على ادعاء "أخضر" — فقط بعد رؤية الناتج الحقيقي بعينك.

### خطأ 2: الوكيل يتخطّى نقطة الوقوف (checkpoint)
**ما حدث:** كنا نراجع A3، والوكيل بنى A4 دون موافقة.
**السبب:** الوكيل يندفع للمهمة التالية بدلًا من انتظار الإذن.
**القاعدة:**
- كل برومبت يجب أن يقول صراحة: "STOP after each task. Do NOT start the next without my explicit 'go'."
- عند التخطّي: أوقفه فورًا، اطلب منه **الاعتراف بالقاعدة** كتابةً، ثم تابع.
- لا تعتمد المهمة التي بناها بالتخطّي إلا بعد مراجعة المهمة الحالية واعتمادها أولًا.

### خطأ 3: الوكيل يطارد العَرَض لا الجذر
**ما حدث:** بق عرض الـ ID بدل الاسم — الوكيل غيّر `defaultValue` → `value` (تخمين منطقي لكنه خطأ). البق بقي.
**السبب:** الوكيل خمّن السبب دون فحص الكود الفعلي.
**القاعدة:**
- بعد محاولة فاشلة واحدة، **لا تتركه يخمّن ثانيةً**. افحص الكود بنفسك:
  `type "<file>" | findstr /n "<keywords>"`
- حدّد الجذر بدقة، ثم أعطِ توجيهًا محدّدًا (السطر بالظبط + الإصلاح)، لا "صلّح البق".
- الفرق: ❌ "يا وكيل صلّح" (تخمين) مقابل ✅ "المشكلة في السطر X، اعمل Y" (دليل).

### خطأ 4: الوكيل يقترح عملًا خارج النطاق
**ما حدث:** الوكيل اقترح "A3 kanban?" بينما البرومبت يحدّد A3 = بروفايل العميل.
**السبب:** الوكيل يبتكر بدل أن يلتزم بالبرومبت.
**القاعدة:**
- في البرومبت، حدّد نطاق كل مهمة بدقة + اكتب "do NOT add anything beyond this".
- عند الاقتراح خارج النطاق: ذكّره بالبرومبت، لا تسمح بالابتكار غير المطلوب.

## القسم 2 — كيف نجعل الوكيل أوضح في تحليل الجذر

### التمييز الثلاثي: compiles ≠ runs ≠ complete
علّم الوكيل (وذكّر نفسك) أن:
- **Compiles** = الكود تجمّع بلا أخطاء صياغة.
- **Runs** = يعمل فعلًا في المتصفح (قد يكون أخضر لكن عالقًا على كاش).
- **Complete** = يحقّق المتطلب فعليًا (تحقّق بصري).
الثلاثة مختلفة. "أخضر" يغطّي الأول فقط.

### اطلب الدليل الخام دائمًا
في كل توجيه إصلاح، أضف:
"Run the REAL build and paste the full raw output, unedited. Do NOT claim green without it."

### بلوك توجيه تحليل الجذر (انسخه في البرومبت)
```
ROOT-CAUSE DISCIPLINE:
- When a fix fails once, STOP guessing. Read the actual code and identify the exact line.
- Report: (1) the exact file:line of the root cause, (2) why it causes the symptom,
  (3) the precise fix. Not a guess — evidence from the code.
- Distinguish "compiles" vs "runs correctly". A green build is not proof it works.
- Paste raw build output as proof. Never report success without it.
```

### استخدم findstr للوصول للجذر بسرعة
بدل قراءة الملف كله:
`type "<file>" | findstr /n "<keyword1> <keyword2>"`
يعطي الأسطر وأرقامها — تصل للموضع بدقة وتوجّه الوكيل بدقة.

## القسم 3 — أخطاء بيئية (ليست خطأ الوكيل)

### الكاش / dev server العالق
**ما حدث:** الـ build أخضر فعلًا، لكن المتصفح أظهر خطأ قديم.
**السبب:** الـ dev server (Turbopack-in-Docker) لا يلتقط التغييرات أحيانًا — يعلق على نسخة قديمة.
**الحل:**
- `docker compose restart app` ثم انتظر دقيقة.
- `Ctrl+Shift+R` في المتصفح (hard refresh).
**القاعدة:** قبل أن تتّهم الكود، استبعد الكاش. شغّل build حقيقي بنفسك — إن كان أخضر، فالمشكلة كاش لا كود.

### worktree + Docker (بورت/اسم مشترك)
**ما حدث:** الـ worktree والـ main يتشاركان نفس الاسم والبورتات (3100/5433) في docker-compose.
**السبب:** `volumes: - .:/app` يربط المجلد الحالي؛ الاسم والبورتات ثابتة.
**الحل (لـ stream واحد):**
- `docker compose down` من المجلد القديم، ثم `docker compose up -d` من الـ worktree.
- البيانات آمنة (named volume `egyglass_db_data` مستقل عن المجلد).
**للمستقبل (streams متوازية):** كل worktree يحتاج compose/بورتات منفصلة — لم يُحلّ بعد.

## القسم 4 — ما نجح (نكرّره)

- **الاختبار قبل البناء:** طلبنا من الوكيل يقرأ البرومبت ويلخّص فهمه قبل أي كود — كشف أنه فهم صح قبل المخاطرة.
- **commit بعد كل مهمة متحقّقة:** A1+A2 محفوظان كنقطة رجوع آمنة قبل A3.
- **مراجعة الكود الأمني بأنفسنا:** قرأنا actions.ts وتأكّدنا من requireRole + zod + ActivityLog.
- **git status لا git diff:** status يُظهر الصورة الكاملة (معدّل + جديد)؛ diff يُظهر المتتبَّع فقط.
- **التحقّق البصري في المتصفح:** كل checkpoint، نفتح الصفحة ونجرّب بأنفسنا — مسك بقّات لم يكشفها الـ build.
- **التوجيه المحدّد عند الإصلاح:** السطر بالظبط + الإصلاح، لا "صلّح".

## الخلاصة — قائمة تحقّق سريعة قبل اعتماد أي checkpoint

1. ☑️ هل شغّلتُ الـ build بنفسي (لا الوكيل)؟ هل الناتج أخضر فعلًا؟
2. ☑️ هل فتحتُ المتصفح وجرّبت الميزة بعينيّ؟
3. ☑️ هل استبعدتُ الكاش (restart + hard refresh) إن ظهر خطأ غريب؟
4. ☑️ هل كل الملفات داخل ملكية الـ stream (git status)؟
5. ☑️ هل الوكيل وقف عند الـ checkpoint ولم يتخطَّ؟
6. ☑️ هل البق (إن وُجد) جذره محدّد، لا مجرّد تخمين؟
