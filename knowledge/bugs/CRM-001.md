---

## 1. Summary

Clicking **Edit** on a customer row in the Customers list, then clicking **Save** without re-typing the optional fields, silently wipes `altPhone`, `address`, `notes` (set to `NULL`) and resets `isRepeat` to `false` in the database. The user is shown a **success** toast (`customers.updated`) — there is no warning, no confirmation, and no undo. The previous values are unrecoverable without a DB backup.

This affects **every** edit performed from the list screen, regardless of role (ADMIN, SALES_MANAGER, SALES_REP).

---

## 2. Root Cause

A two-layer defect: the list query never loads the four optional fields, so the client opens the Edit dialog with hard-coded empty defaults; the server then accepts those empty defaults as a valid update and overwrites the real DB values.

### Layer 1 — Service `getCustomers` does not select the four fields

`src/lib/services/customers.ts:48-62` — the `prisma.customer.findMany` `select` clause returns only:

```
id, name, phone, type, source, stage, ownerId, coveredById, owner.name
```

`altPhone`, `address`, `notes`, `isRepeat` are **not** selected. The `CustomerRow` interface (`services/customers.ts:4-16`) correspondingly omits them:

```ts
export interface CustomerRow {
  id: string;
  name: string;
  phone: string;
  type: string;
  source: string;
  stage: string;
  ownerName: string | null;
  ownerId: string | null;
  coveredById: string | null;
  coveredByName: string | null;
  isCoverage: boolean;
}
```

### Layer 2 — Client `openEdit` resets the form with hard-coded defaults

`src/app/(dashboard)/customers/customers-client.tsx:150-164`:

```ts
function openEdit(customer: CustomerRow) {
  setEditingCustomer(customer);
  reset({
    name: customer.name,
    phone: customer.phone,
    altPhone: "",          // ← not in CustomerRow, hard-coded
    type: customer.type,
    source: customer.source,
    address: "",           // ← not in CustomerRow, hard-coded
    notes: "",             // ← not in CustomerRow, hard-coded
    isRepeat: false,       // ← not in CustomerRow, hard-coded
    ownerId: customer.ownerId ?? undefined,
  });
  setOpen(true);
}
```

The author worked around the missing fields by resetting them to `""` / `false` instead of referencing `customer.altPhone` etc. (which would not even compile against `CustomerRow`). The form therefore opens with empty inputs for `altPhone`, `address`, `notes`, and an unchecked `isRepeat` checkbox — **regardless of what is stored in the DB**.

### Layer 3 — Server action + service treat empty defaults as a real update

`src/app/(dashboard)/customers/actions.ts:22-33` — `updateSchema`:

```ts
const updateSchema = z.object({
  id: z.string(),
  name: z.string().min(1, "errors.required").optional(),
  phone: z.string().min(1, "errors.required").optional(),
  altPhone: z.string().optional(),     // "" is a valid string
  type: customerTypeEnum.optional(),
  source: customerSourceEnum.optional(),
  address: z.string().optional(),      // "" is a valid string
  notes: z.string().optional(),        // "" is a valid string
  isRepeat: z.boolean().optional(),    // false is a valid boolean
  ownerId: z.string().optional(),
});
```

`""` passes `z.string().optional()` and `false` passes `z.boolean().optional()`. Validation succeeds.

`src/lib/services/customers.ts:146-181` — `updateCustomer` writes any field that is `!== undefined`:

```ts
if (input.altPhone !== undefined) { data.altPhone = input.altPhone || null; changed.push("altPhone"); }
if (input.address  !== undefined) { data.address  = input.address  || null; changed.push("address");  }
if (input.notes    !== undefined) { data.notes    = input.notes    || null; changed.push("notes");    }
if (input.isRepeat !== undefined) { data.isRepeat = input.isRepeat;          changed.push("isRepeat"); }
```

Because the client always sends `altPhone: ""`, `address: ""`, `notes: ""`, `isRepeat: false`, all four conditions fire on every edit. `"" || null` evaluates to `null`, so the three string fields become `NULL`; `isRepeat` becomes `false`. Prisma `update` writes them.

### Why the success toast is a lie

`customers-client.tsx:182-193` — `updateCustomerAction` returns `{ success: true, data: result.data }`, the client calls `toast.success(t("customers.updated"))` and replaces the row. There is no comparison between old and new values, so the wipe is reported as a successful update.

### Audit trail is also corrupted

`services/customers.ts:170-178` writes an `ActivityLog` row with `action: "CUSTOMER_UPDATED"` and `details: { changes: ["altPhone","address","notes","isRepeat", ...] }`. The audit log therefore **records the data loss as a legitimate change** — there is no way to distinguish a real edit from a wipe after the fact.

## 6. Database Impact

Direct writes on every Edit+Save from `/customers`:

| Column | Type | Before | After |
|---|---|---|---|
| `Customer.altPhone` | `String?` | any value | `NULL` |
| `Customer.address` | `String?` | any value | `NULL` |
| `Customer.notes` | `String?` | any value | `NULL` |
| `Customer.isRepeat` | `Boolean @default(false)` | any value | `false` |
| `Customer.updatedAt` | `DateTime @updatedAt` | — | bumped to now |
| `ActivityLog` (new row) | — | — | `action=CUSTOMER_UPDATED`, `details.changes` lists the four fields — **false positive audit entry** |

**No soft-delete, no versioning, no history table.** The previous values are unrecoverable unless a DB backup or PITR exists. There is no Prisma transaction wrapping the update + audit log, but that is not the cause of the loss — it merely means a partial-failure rollback is also not handled.

The wipe is **idempotent**: once a customer has been edited once from the list, all four fields are already NULL/false, so subsequent edits appear to "work fine" — masking the bug from casual testing.

## 7. Why Data Is Lost (mechanism, step by step)

1. User opens `/customers`. `getCustomers` runs the `findMany` with a `select` that **excludes** `altPhone`, `address`, `notes`, `isRepeat`.
2. User clicks **Edit** on a row. `openEdit(customer)` receives a `CustomerRow` that has **no** `altPhone`/`address`/`notes`/`isRepeat` properties.
3. `openEdit` calls `reset({ altPhone: "", address: "", notes: "", isRepeat: false, ... })` because it cannot read those fields from `customer` (they would be a TypeScript error against `CustomerRow`).
4. The Edit dialog renders with empty inputs for those four fields. The user does not see the existing values and is not warned.
5. User clicks **Save**. `onSubmit(formData)` sends `{ id, name, phone, altPhone: "", address: "", notes: "", isRepeat: false, type, source, ownerId }` to `updateCustomerAction`.
6. `updateSchema.safeParse` succeeds — `""` and `false` are valid.
7. `updateCustomer` sees all four fields as `!== undefined`, so it writes `altPhone = null`, `address = null`, `notes = null`, `isRepeat = false`.
8. Prisma `update` commits. `toRow(customer)` returns the updated row (still without the four fields, since `CustomerRow` doesn't carry them).
9. Client replaces the row in state and shows `toast.success("customers.updated")`.

**Root cause in one sentence:** The list query and `CustomerRow` type were narrowed for performance/display, but the Edit dialog reuses that narrowed row as the source of truth for the form, and the update path accepts the resulting empty defaults as a valid write — so the form silently overwrites DB fields it never loaded.

## 9. Proposed Fix (NOT applied — investigation only)

**Recommended: Option A (client + service select).**

1. **Extend `CustomerRow`** (`src/lib/services/customers.ts:4-16`) to include:
   ```ts
   altPhone: string | null;
   address: string | null;
   notes: string | null;
   isRepeat: boolean;
   ```
2. **Extend the `getCustomers` select** (`services/customers.ts:51-61`) to fetch `altPhone`, `address`, `notes`, `isRepeat`.
3. **Update `toRow`** (`services/customers.ts:93-107`) to map the four fields.
4. **Fix `openEdit`** (`customers-client.tsx:150-164`) to populate from the row:
   ```ts
   reset({
     name: customer.name,
     phone: customer.phone,
     altPhone: customer.altPhone ?? "",
     type: customer.type,
     source: customer.source,
     address: customer.address ?? "",
     notes: customer.notes ?? "",
     isRepeat: customer.isRepeat,
     ownerId: customer.ownerId ?? undefined,
   });
   ```

**Optional hardening (defense in depth):**

5. In `updateCustomer`, treat `""` for an optional string field as "no change" only when the caller explicitly passes `undefined` — i.e. rely on the client omitting unchanged fields. This requires changing `onSubmit` to build a partial payload of dirty fields (rhf `formState.dirtyFields`) rather than submitting the whole form. Lower priority than Option A.
6. Add a server-side guard: if `updateCustomer` receives `altPhone === ""` AND `address === ""` AND `notes === ""` AND `isRepeat === false` simultaneously, log a warning — this pattern strongly indicates a default-reset bug, not a real edit.

**Do NOT** "fix" this by hiding the Edit button or making the fields read-only on the list — that masks the bug and breaks the UAT test case.

4. Observe: the `altPhone`, `address`, `notes` inputs are empty and the `isRepeat` checkbox is unchecked — **this is the bug visible at open time**.
