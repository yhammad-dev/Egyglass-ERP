## onDelete Behavior — Full Table

| Relation (child → parent) | onDelete | Justification |
|---|---|---|
| `QuotationRequest.customer` → `Customer` | `Restrict` | Cannot delete a customer who has active TEC jobs |
| `QuotationRequest.quotation` → `Quotation` | `Restrict` | Quotation is the source of truth; deleting it must be blocked while a TEC job exists |
| `QuotationRequest.inspectionRequest` → `InspectionRequest` | `SetNull` | Inspection may be soft-deleted or reassigned; TEC job survives — `inspectionRequestId` becomes null |
| `QuotationRequest.engineer` → `User` | `SetNull` | Engineer may be deactivated; job stays, just unassigned |
| `QuotationRequest.salesOwner` → `User` | `SetNull` | Sales rep may leave; job stays, owner becomes null |
| `QuotationRequest.inspectionOwner` → `User` | `SetNull` | Inspection owner may be reassigned; job stays, field becomes null |
| `Drawing.quotationRequest` → `QuotationRequest` | `Cascade` | Drawings have no meaning without their parent job; delete together |
| `Drawing.uploadedBy` → `User` | `Restrict` | Audit trail: cannot delete a user who has uploaded drawings |
| `Drawing.approvedBy` → `User` | `SetNull` | Approver may be deactivated; record survives, approval attribution becomes null |
| `ExtraItem.manufacturingOrder` → `ManufacturingOrder` | `Restrict` | Extra items have cost implications; must not be silently deleted with order |
| `ExtraItem.createdBy` → `User` | `Restrict` | Audit trail: cannot delete user who created cost-bearing extra items |

## 1. Model: `QuotationRequest`

**Purpose:** Technical Office job card. One record per job handed off from Sales/Inspection
to the TEC. Drives the kanban board (Phase 1 TEC-10) and anchors drawings, re-pricing, and
extra-item workflows.

```prisma
model QuotationRequest {
  // ── Identity ──────────────────────────────────────────────────────────────
  id                  String           @id @default(cuid())
  code                String           @unique @db.VarChar(50)
  // Generated by server action on creation: "TEC-YYYY-NNNN".
  // Unique constraint — one canonical reference per job.

  // ── Kanban state ──────────────────────────────────────────────────────────
  status              TecJobStatus     @default(NEW)
  // Maps to kanban columns: NEW / IN_PROGRESS / ON_HOLD / DONE.

  technicalRoute      TechnicalRoute   @default(PROJECTS)
  // DEDICATED field — NOT derived from CustomerSource.
  // Set explicitly by TEC on job-card creation.
  // Confirmed values: PROJECTS | SOCIAL_MEDIA.

  // ── Content ───────────────────────────────────────────────────────────────
  summary             String?          @db.Text
  // Brief description of the job as received from Sales.

  notes               String?          @db.Text
  // Internal TEC notes; not visible to Sales.

  // ── Source references ─────────────────────────────────────────────────────
  customerId          String
  // Denormalized for fast customer-profile lookup; kept in sync at creation only.

  quotationId         String           @unique
  // @unique: one TEC job per source quotation.
  // Revised quotations are separate Quotation rows (via Quotation.previousQuotationId).

  inspectionRequestId String?
  // Explicit FK to the InspectionRequest that triggered this job, if any.
  // Null for direct-quotation paths (no inspection).
  // onDelete: SetNull (see onDelete table above).

  // ── Three distinct User roles ─────────────────────────────────────────────
  engineerId          String?
  // The TEC engineer responsible for execution of this job.
  // Nullable: may be unassigned at creation.
  // onDelete: SetNull.

  salesOwnerId        String?
  // The sales rep who owns the customer.
  // Copied from Customer.ownerId at job-card creation and stored explicitly
  // so it is stable even if the customer is later reassigned.
  // onDelete: SetNull.

  inspectionOwnerId   String?
  // The User who was assignee on the linked InspectionRequest.
  // Copied at job-card creation; stored separately from inspectionRequestId
  // so the name is available even if the inspection record is soft-deleted.
  // onDelete: SetNull.

  // ── Timestamps ────────────────────────────────────────────────────────────
  createdAt           DateTime         @default(now())
  updatedAt           DateTime         @updatedAt
  deletedAt           DateTime?
  // Soft-delete support, matching Customer / Quotation convention.

  // ── Relations ─────────────────────────────────────────────────────────────
  customer            Customer            @relation(
                                            fields:     [customerId],
                                            references: [id],
                                            onDelete:   Restrict
                                          )

  quotation           Quotation           @relation(
                                            fields:     [quotationId],
                                            references: [id],
                                            onDelete:   Restrict
                                          )

  inspectionRequest   InspectionRequest?  @relation(
                                            fields:     [inspectionRequestId],
                                            references: [id],
                                            onDelete:   SetNull
                                          )

  engineer            User?               @relation(
                                            name:       "TecEngineer",
                                            fields:     [engineerId],
                                            references: [id],
                                            onDelete:   SetNull
                                          )

  salesOwner          User?               @relation(
                                            name:       "TecSalesOwner",
                                            fields:     [salesOwnerId],
                                            references: [id],
                                            onDelete:   SetNull
                                          )

  inspectionOwner     User?               @relation(
                                            name:       "TecInspectionOwner",
                                            fields:     [inspectionOwnerId],
                                            references: [id],
                                            onDelete:   SetNull
                                          )

  drawings            Drawing[]
  // One-to-many: all drawings attached to this job.

  // ── Indexes ───────────────────────────────────────────────────────────────
  @@index([status])
  // TEC-10 kanban: group jobs by column (NEW / IN_PROGRESS / ON_HOLD / DONE).

  @@index([technicalRoute])
  // TEC-10 kanban: filter by tab (PROJECTS / SOCIAL_MEDIA).

  @@index([status, technicalRoute])
  // TEC-10 dual-filter: kanban renders one tab+column combination at a time;
  // composite index avoids two separate scans.

  @@index([customerId])
  // Customer profile page: list all TEC jobs for a given customer.

  @@index([engineerId])
  // "My jobs" view for a logged-in TEC engineer.

  @@index([inspectionRequestId])
  // Inspection detail: show whether an inspection has a linked TEC job.

  @@map("quotation_requests")
}
```

**Back-relations to add on existing models (Prisma-level only — no DB columns):**

```prisma
// On User — three NAMED back-relations:
tecJobsAsEngineer        QuotationRequest[]  @relation("TecEngineer")
tecJobsAsSalesOwner      QuotationRequest[]  @relation("TecSalesOwner")
tecJobsAsInspectionOwner QuotationRequest[]  @relation("TecInspectionOwner")

// On Customer:
quotationRequests        QuotationRequest[]

// On Quotation:
quotationRequest         QuotationRequest?
// Optional: most quotations will not have a TEC job at creation time.

// On InspectionRequest:

// Optional: not every inspection leads to a TEC job.
```

## 2. Model: `Drawing`

**Purpose:** Technical drawing or document uploaded by the TEC for a specific job.
Supports categorized file storage, embedded viewing (PDF/JPG), download-only links (DWG),
and a two-state approval workflow (unapproved → approved).

```prisma
model Drawing {
  // ── Identity ──────────────────────────────────────────────────────────────
  id                   String           @id @default(cuid())

  // ── Classification ────────────────────────────────────────────────────────
  category             DrawingCategory
  // See DrawingCategory enum below for all values.

  fileType             DrawingFileType
  // PDF + JPG: embedded viewer. DWG: download/link only.

  // ── File storage (matches Document / InspectionPhoto convention exactly) ──
  filename             String           @db.VarChar(300)
  // Server-assigned name on disk: "drw-<uuid>.<ext>"

  originalName         String           @db.VarChar(300)
  // Original filename as provided by the uploader; displayed in UI.

  mimeType             String           @db.VarChar(100)
  // e.g. "application/pdf", "image/jpeg", "application/acad"

  url                  String           @db.VarChar(1000)
  // Public path served by Next.js: "/uploads/drawings/<filename>"

  sizeBytes            Int
  // File size in bytes; displayed as "XX KB / MB" in UI.

  // ── Optional metadata ─────────────────────────────────────────────────────
  label                String?          @db.VarChar(200)
  // Human-readable label (e.g. "خطة الطابق الأول"). Falls back to originalName in UI.

  notes                String?          @db.Text
  // Internal TEC notes about this drawing.

  revision             String?          @db.VarChar(20)
  // Revision marker entered by uploader: "Rev A", "Rev 2", etc.

  // ── Parent job ────────────────────────────────────────────────────────────
  quotationRequestId   String
  // onDelete: Cascade — drawings are meaningless without their parent job.

  // ── Upload provenance ─────────────────────────────────────────────────────
  uploadedById         String
  // onDelete: Restrict — cannot delete a user who has uploaded drawings (audit trail).

  // ── Approval workflow ─────────────────────────────────────────────────────
  approvedById         String?
  // Null = not yet approved.
  // Populated atomically with approvedAt by the approval server action.
  // RBAC: restricted to Role.TEC_APPROVER (⚠️ see §Q2 — name pending).
  // onDelete: SetNull — approver deactivation does not invalidate the approval record.

  approvedAt           DateTime?
  // Null until approved. Set together with approvedById in one transaction.

  // ── Timestamps ────────────────────────────────────────────────────────────
  createdAt            DateTime         @default(now())
  updatedAt            DateTime         @updatedAt

  // ── Relations ─────────────────────────────────────────────────────────────
  quotationRequest     QuotationRequest @relation(
                                          fields:     [quotationRequestId],
                                          references: [id],
                                          onDelete:   Cascade
                                        )

  uploadedBy           User            @relation(
                                          name:       "DrawingUploader",
                                          fields:     [uploadedById],
                                          references: [id],
                                          onDelete:   Restrict
                                        )

  approvedBy           User?           @relation(
                                          name:       "DrawingApprover",
                                          fields:     [approvedById],
                                          references: [id],
                                          onDelete:   SetNull
                                        )

  // ── Indexes ───────────────────────────────────────────────────────────────
  @@index([quotationRequestId])
  // Primary: load all drawings for a given TEC job.

  @@index([quotationRequestId, category])
  // Drawing library tabs: filter by category within a job without full scan.

  @@index([approvedById])
  // Approval dashboard: WHERE approvedById IS NULL surfaces pending drawings.

  @@map("drawings")
}
```

**Back-relations to add on User (additive only):**

```prisma
uploadedDrawings   Drawing[]  @relation("DrawingUploader")
approvedDrawings   Drawing[]  @relation("DrawingApprover")
```

## 3. Model: `ExtraItem`

**Purpose:** Additional work items added by the TEC to an existing manufacturing order
(chamfer, welding, accessories, transport, sanding). On creation the server action sends
a Notification to all users with Role.PROCUREMENT (Shukri), using the existing
Notification model and notification flow.

```prisma
model ExtraItem {
  // ── Identity ──────────────────────────────────────────────────────────────
  id                   String        @id @default(cuid())

  // ── Classification ────────────────────────────────────────────────────────
  type                 ExtraItemType
  // See ExtraItemType enum below for all values.

  // ── Detail ────────────────────────────────────────────────────────────────
  description          String?       @db.VarChar(500)
  // Free-text detail. Optional for standard types; recommended for EXTRA_ACCESSORY
  // where the specific accessory name must be captured.

  qty                  Decimal?      @db.Decimal(10, 2)
  // Quantity. Nullable: OUT_OF_CAIRO_TRANSPORT is typically a flat item with no qty.

  unitCost             Decimal?      @db.Decimal(12, 2)
  // Cost per unit at time of entry — informational for procurement.
  // Pricing authority stays with Sales; this is a reference figure only.

  notes                String?       @db.Text
  // Additional context for the Procurement team.

  // ── Manufacturing order reference ─────────────────────────────────────────
  manufacturingOrderId String
  // onDelete: Restrict — extra items carry cost implications;
  // manufacturing orders cannot be deleted while extra items exist on them.

  // ── Provenance ────────────────────────────────────────────────────────────
  createdById          String
  // onDelete: Restrict — audit trail; cannot delete user who created cost-bearing items.

  createdAt            DateTime      @default(now())
  updatedAt            DateTime      @updatedAt

  // ── Relations ─────────────────────────────────────────────────────────────
  manufacturingOrder   ManufacturingOrder @relation(
                                            fields:     [manufacturingOrderId],
                                            references: [id],
                                            onDelete:   Restrict
                                          )

  createdBy            User               @relation(
                                            name:       "ExtraItemCreator",
                                            fields:     [createdById],
                                            references: [id],
                                            onDelete:   Restrict
                                          )

  // ── Indexes ───────────────────────────────────────────────────────────────
  @@index([manufacturingOrderId])
  // Load all extra items for a manufacturing order (order detail page).

  @@index([type])
  // Procurement filter: show all items of a given type across all orders.

  @@map("extra_items")
}
```

**Back-relations to add on existing models (additive only):**

```prisma
// On User:
createdExtraItems  ExtraItem[]  @relation("ExtraItemCreator")

// On ManufacturingOrder:
extraItems         ExtraItem[]
```

## 5. Role Enum — Additive Additions

**Context:**
- `REVIEW` (already in schema) is used by the Inspection Manager reviewing
  inspection results per PRC-11. It is a **distinct** role from drawing approval.
- Drawing approval (TEC-13) is performed by designated approvers ("Nouran / Amr").
  These are **individual people** assigned a role — they are NOT named in the schema.

## Q2 — Drawing Approver Role: Options A / B / C

**Constraint:** `REVIEW` (Inspection Manager, PRC-11) and TEC drawing approval
(TEC-13) are **DISTINCT** roles. Do not collapse them into one.

---

### Option A — Add `TEC_APPROVER` as a new Role enum value

```prisma
enum Role {
  // ... existing values ...
  TECHNICAL_OFFICE   // TEC engineers
  TEC_APPROVER       // Drawing approvers (Nouran / Amr assigned here)
}
```

- Schema change: +2 new Role values (`TECHNICAL_OFFICE` + `TEC_APPROVER`)
- RBAC: server actions check `role === "TEC_APPROVER"`
- Pros: explicit, future-proof, zero ambiguity
- Cons: one additional enum value to manage

---

### Option B — Reuse `ADMIN` or `SALES_MANAGER` for drawing approval

```prisma
enum Role {
  // ... existing values ...
  TECHNICAL_OFFICE   // TEC engineers only
  // No new approver role — approval gated on Role.ADMIN or Role.SALES_MANAGER
}
```

- Schema change: +1 new Role value (`TECHNICAL_OFFICE` only)
- RBAC: server actions check `role === "ADMIN" || role === "SALES_MANAGER"`
- Pros: no new enum value; Amr (SALES_MANAGER) and Youssif (ADMIN) already exist
- Cons: grants drawing approval to **all** Admins and all Sales Managers —
  wider than intended if those roles expand in future

---

### Option C — Add both `TECHNICAL_OFFICE` and `TEC_APPROVER` as separate values *(recommended)*

```prisma
enum Role {
  // ... existing values ...
  TECHNICAL_OFFICE   // TEC engineers: create jobs, upload drawings, add extra items, re-price
  TEC_APPROVER       // Drawing approvers only: approve/reject drawings (TEC-13)
}
```

- Schema change: +2 new Role values
- RBAC: engineer actions check `role === "TECHNICAL_OFFICE"`;
  approval action checks `role === "TEC_APPROVER"`
- Engineers ≠ Approvers: a TEC engineer cannot approve their own drawings
- Pros: most precise, enforces separation of duties, roles stay abstract
- Cons: Youssif must assign `TEC_APPROVER` to exactly the right people in the User table

**Recommendation: Option C.**
Reasoning: Role.REVIEW (inspection) and Role.TEC_APPROVER (drawings) are already
parallel to this pattern. Separation of duties (engineer ≠ approver) is a real
operational requirement for the drawing approval flow. Option B grants too broad
access. Option C costs one extra enum value and zero other complexity.

**Awaiting your selection (A / B / C) before this draft is applied.**
