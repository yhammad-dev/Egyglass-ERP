## 1. هوية المشروع + Stack

- **المشروع:** EgyGlass ERP — نظام تشغيلي متكامل لشركة زجاج معماري (القاهرة).
- **الدور البشري:** يوسف = المعتمد الوحيد للـ schema والـ migrations والقرارات. الوكيل ينفّذ، يوسف يراجع ويعتمد.
- **Stack:** Next.js **15.3.4** (App Router, Turbopack) · Prisma **6.19.3** · PostgreSQL 16 · Auth.js v5 · shadcn/ui v4 · Tailwind v4 (RTL) · next-intl v4 · react-hook-form + Zod · TanStack Table · Docker Compose.
  - ⚠️ الواقع = **Next 15.3.4** (ليس 16). لا تُرقِّ Prisma إلى 7 أثناء هذه المرحلة.
  - ⚠️ **تحذير `NODE_ENV` غير القياسي وقت build** لا يزال يظهر (جذر الدرس 5) → يسبب رسائل custom Document / missing `<Html>`. **يجب إصلاحه قبل UAT** (`NODE_ENV=production` وقت build).
- **المنافذ:** التطبيق `3100` · قاعدة البيانات `5433` (داخل Docker: `db:5432`) · Named volume: `egyglass_db_data`.
- **Repo:** `yhammad-dev/Egyglass-ERP` (master) · git worktrees لعزل الـ streams.

### البيئة (مهم)
- **الآن: تطوير محلي على جهاز يوسف عبر Docker فقط.**
- **UAT (EC2) = رفعة واحدة لاحقًا — لا تلمس إعدادات UAT الآن.**

## 2. المبادئ الحاكمة (غير قابلة للتفاوض)

1. **بالدليل لا بالحدس.** ادعاء الوكيل ("done/complete/resolved") ليس دليلًا.
2. **التحقق = build GREEN + route HTTP 200** على صفحة مُصادَقة فعليًا — **ليس 307** (redirect للـ auth)، وليس مجرد compile.
   ميّز دائمًا: **Compiles ≠ Runs ≠ Complete**.
3. **Config-driven:** كل قاعدة عمل (نِسب خصم، عتبات، مواضع بوابات) تعيش في **البيانات لا الكود**. **ممنوع رقم عمل صلب في الكود.**
4. **Role-based — بلا أسماء في الكود.** الموظف يمشي بالدور؛ الدور هو الباقي.
5. **الـ Schema مجمّد.** يوسف وحده يطبّق الـ migrations داخل Docker. لو احتجت تغييرًا → **قف، اكتب change-request، انتظر**. لا تعدّل schema بنفسك.
6. **Checkpoints:** قف عند كل نقطة، بلّغ بالدليل، انتظر موافقة صريحة قبل المتابعة. لا تنطلق للأمام.
7. **FIX LIMIT = محاولتان** لكل خطأ، ثم قف وصعّد. لا تطارد الأعراض — دوّر على الجذر.
8. **تحقّق بصري قبل الـ commit** ("بعيني قبل ما نعمل commit").
9. **القرارات تُقفل قبل تشغيل الوكيل.** لا تُدخِل غموضًا أثناء التنفيذ.
10. **كل server action يبدأ بـ `requireRole`.** كل mutation تكتب `ActivityLog`. كل نص للمستخدم عبر `t()` (المفتاح موجود في **ملفَّي اللغة معًا** ar + en).
11. **Reconnaissance قبل الكود (Phase X.0):** حدّد المكوّنات والأنماط الموجودة لإعادة استخدامها قبل كتابة أي شيء.
12. **طابِق التحقق بقابلية التراجع (الدرس 12):** الخطوات القابلة للرجوع → بثقة معقولة. غير القابلة (schema، حذف بيانات) → دليل مرئي وتأنٍّ.

## 3. الأدوار (RBAC — بالوظيفة لا بالاسم)

> ⚠️ **الأسماء الفعلية من الكود (متحقَّق P0.0) — استخدمها حرفيًا في `requireRole`.** لا يوجد `QUALITY_REVIEW` — استخدم `REVIEW`.

## 4. القرارات المعتمدة (14 — مختصرة)

- **W-01** طلبات التسعير: Single Entry Point + `requestSource` enum. لا تسعير خارج النظام.
- **W-02** التعديل بعد المعاينة: مسار واحد يتقارب عند المكتب الفني (مالك إعادة التسعير). المبيعات تُخطَر دائمًا.
- **W-03** المشروعات: اكتساب موحد عبر المبيعات · توجيه داخلي بالمصدر · كريم بعد التعاقد (app-layer gate: APPROVED فقط).
- **W-04** بطاقة الإكسسوار: TEC ينشئ · INS يؤكد المقاسات (`confirmedByInspection`) · PRC ينفّذ التوريد دون تعديل.
- **W-05 / R-01** اعتماد الرسومات: بوابتان متمايزتان + CEO مشروط. Gate1 `TEC_APPROVER` · Gate2 `INSPECTION_MANAGER` (verify) · Gate3 CEO (عتبة config) · دور `REVIEW` (محمد حسام) موضعه في السلسلة = config. المهندس لا يعتمد رسمته.
- **W-06** بدل الكسر: Fast-track (يتخطى Gate1، الرسمة معتمدة) + `parentOrderId` + `faultType` + إشعار للمسؤول.
- **W-07** استطلاع الرضا: event-driven عند `INSTALLED_FINAL` → إشعار المبيعات → استطلاع بعد `satisfactionSurveyDelayDays` (config=3).
- **R-02** كتابة عملاء الزملاء: مسموح + logging كامل + لوحة رؤية للمدير (soft-control).
- **R-03** قراءة المالية: project-scoped (least privilege). كريم/نوران = مشاريعهم فقط.
- **R-04** التركيبات: حساب لكل قائد فريق + مدير override (هو التغطية). لا deputy منفصل.
- **P-01** الـ Price List: **استيعابه داخل النظام** versioned (SCR-014، قرار schema منفصل). محرك الخصم يحتاج السعر الأساسي داخل النظام.
- **P-02** حالات عرض السعر: state machine كامل `DRAFT→SENT→PENDING_INSPECTION→POST_INSPECTION_REVISED→SENT_REVISED→ACCEPTED→CONTRACTED` + REJECTED/EXPIRED/PRICE_CHANGE_INVALIDATED.
- **P-03** الضمان/الصيانة: ضمان = نص تعاقدي ثابت (بلا سعر) · صيانة مدفوعة = بند اختياري مُسعَّر حسب الحالة.

## 5. الأسئلة المفتوحة → تُمتَص كإعدادات (لا تحجب البناء)

- **P-04 (الخصم):** `discountBasePct = 18` موجود ومثبّت · `discountMaxReqPct = 25` سقف مطلق موجود · **يُضاف عمود واحد** `managerApprovalCeilingPct` (الحد الذي يعتمده SALES_MANAGER؛ فوقه → CEO) — قيمته تبقى فارغة حتى يحددها عمرو · `ceoDrawingApprovalThreshold` عمود جديد أيضًا (عتبة Gate3).
- **R-05 (المراجعة):** دور `REVIEW` **موجود بالفعل** (محمد حسام). موضعه في سلسلة الاعتماد `reviewGatePosition` = **config**، ينتظر التوضيح. لا تُنشئ دورًا جديدًا.

> **عند وصول الردود:** تعديل قيمة إعداد فقط — **لا كود، لا migration، لا إعادة اختبار بنية.**

## 6. أنماط الكود الإلزامية

- `requireRole(...)` في أول كل server action.
- `ActivityLog` عند كل mutation (من، ماذا، متى).
- `t()` لكل نص — المفتاح في `ar` و `en` معًا (وإلا build يكسر).
- **shadcn form غير موجود في v4** → استخدم `react-hook-form + zod` مباشرة عبر helper موحّد `FieldError`.
- `type="button"` على كل زر داخل `<form>` (وإلا auto-submit).
- triggers الـ enum/select تُظهر labels مترجمة.
- احترم حدود ملكية الملفات لكل stream.
- راجع الكود الموجود قبل استبداله (لا تفترض).

## 8. حالة الـ Schema

- مجمّد على `schema-phase1-done` (79b488d) · `SCR-010` (TEC: QuotationRequest, Drawing, ExtraItem + TECHNICAL_OFFICE/TEC_APPROVER) · `SCR-011` مطبّقان.
- SAL gaps مقفولة (`sal-gaps-closed` e5b4ff8): خصم هرمي + كاش باك، أسباب رفض، username في الـ activity، WhatsApp، row-level security لـ SALES_REP، VIP، استطلاع ما بعد التركيب.
- INS: G1/G2/G3 منفّذة · G4 (`siteReadiness`, SCR-012) معلّق.
- **يوسف وحده يطبّق أي migration، داخل Docker.**

### واقع الـ schema (متحقَّق P0.0 — يغلب أي افتراض)
- **`SystemSettings` = أعمدة ثابتة typed بنمط singleton (id="singleton") — ليس key-value.** أي مفتاح إعداد جديد = **schema change** (SCR-013).
  - الأعمدة الحالية: `discountBasePct=18` · `discountMaxReqPct=25` (حد واحد) · `factorMinimum=1.5` · `vatPct=14` · `quotationValidDays=3` · `cashbackActive` · `cashbackStartDate` · logo/name/updatedBy/updatedAt.
- **نماذج موجودة بالفعل:** `PriceListItem` · `PricingFactor` · `CashbackTier` → **P-01 منفّذ إلى حد كبير**؛ تحقّق منها قبل افتراض SCR-014.
- الرقم 18 في الكود = **fallback default فقط**؛ القيمة الحقيقية من DB. **لا accessor مركزي** — كل ملف له `getSettings()` خاص (يجب توحيده في `lib/config.ts`).

## 9. سير العمل الرئيسي (مؤكد)

```
المبيعات → (طلب تسعير) → المكتب الفني → (عرض سعر) → المبيعات
المبيعات → (تحويل معاينة) → المعاينات → (مقاسات) → المكتب الفني
المكتب الفني → (إعادة تسعير) → المبيعات → (تعاقد) → الحسابات + المكتب الفني + المشروعات
المكتب الفني → (رسم تنفيذي + اعتماد) → المخازن → (تصنيع) → التركيبات → (إغلاق) → المشروعات + المبيعات
```

## 12. متطلبات التصميم / الواجهة (P3 — طبقة تطبيق)

- **الشعار (Logo) عالمي وديناميكي:** يظهر في **كل الشاشات** والمستندات (رأس الصفحة، العقد، الفاتورة). يُقرأ من `SystemSettings.companyLogoUrl` **الموجود بالفعل** — **لا حاجة إلى schema جديد**.
  - تعديله يتم من **شاشة الأدمن (عمرو) فقط** (`requireRole('ADMIN')` + `ActivityLog`)، وينعكس فورًا في كل الشاشات (shared header component يقرأ القيمة).
- **تصميم العقد والفاتورة:** يُنمذَج على **عينة تقدّمها الشركة** (معلّقة حاليًا — تنتظر الملف). الشعار الديناميكي يظهر فيهما أيضًا. بند P3.
- **RTL عربي أساسي** عبر next-intl؛ كل نص عبر `t()`.

> **ملاحظة:** لا تصمّم العقد/الفاتورة قبل وصول عينة الشركة — تجنّب إعادة العمل.
